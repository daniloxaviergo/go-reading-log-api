### 4. Code Patterns

**Directory structure pattern (Clean Architecture):**
```
go-reading-log-api-next/
├── cmd/                     # Application entry points
│   └── server.go           # Main, creates and starts server
├── internal/               # Private application code
│   ├── api/                # API layer
│   │   └── v1/            # API version 1
│   │       ├── handlers/  # HTTP handlers
│   │       └── middleware/ # HTTP middleware
│   ├── adapter/           # External service adapters
│   │   └── postgres/      # PostgreSQL implementations
│   ├── config/            # Configuration loading
│   ├── domain/            # Domain models and DTOs
│   │   ├── models/        # Domain entities
│   │   └── dto/           # Data Transfer Objects
│   ├── repository/        # Repository interfaces
│   └── logger/            # Logger setup
├── test/                  # Test utilities and helpers
├── go.mod                 # Module definition
├── go.sum                 # Dependency checksums
└── .env.example           # Environment template
```

**Naming conventions:**
- Package names = directory names (lowercase, singular)
- Interface names end with `Repository` or `Service`
- Implementation names = Interface + `Impl`
- Handler functions = HTTP method + resource name (e.g., `ProjectsHandler`, `LogsHandler`)
- Error handling: returning `{"error": "message"}` format

**Integration patterns:**
- Use `net/http` handlers with clean function signatures
- Repository pattern: interfaces in `repository/`, implementations in `adapter/`
- Configuration via struct loaded from environment variables
- Logger passed through context or as dependency
